# crypton 프로덕션 준비 평가 및 로드맵

> 작성일 2026-06-06 · 대상 커밋 `f7e16f8` · 평가 기준 **OWASP ASVS 4.0.3**, 12-Factor, SOC 2 공통 통제
> 이 문서는 코드 실측에 근거한 **정직한 보안/운영 성숙도 평가**입니다. 인증된 외부 감사가 아니라 내부 전문가 매핑입니다.

---

## 0. 한 줄 결론

| 관점 | 점수 | 평가 |
|------|------|------|
| **발명(토큰 회전)의 PoC 완성도** | **~90%** | 핵심 메커니즘이 견고하고 테스트로 검증됨 |
| **OWASP ASVS L1 (앱 보안 최소 기준)** | **~25%** | 프로덕션 보안 제품으로는 초기 단계 |
| **OWASP ASVS L2 (유료·민감 데이터 권장 기준)** | **~15%** | 상용 출시 기준에는 크게 미달 |
| **프로덕션 SaaS 종합 준비도** (보안+영속성+운영+컴플라이언스) | **~20%** | 배포 가능 상태 아님 |

**요지:** crypton은 *발명의 동작을 증명하는 우수한 PoC*이지만 *프로덕션급 소스는 아닙니다.* 단일 최대 차단 요인은 **인증 부재**(누구나 요청 본문에 임의의 `userId`를 넣어 타인의 권한을 행사 가능)와 **전송 암호화(TLS) 부재**(토큰·콘텐츠 키가 평문 전송)입니다. 이 둘은 %로 환산되는 “부족”이 아니라 **배포 불가**를 의미하는 이진적 결격입니다.

- **OWASP ASVS 4.0.3** — 웹/API 보안 검증의 사실상 산업 표준. 14개 챕터(V1–V14)를 L1(최소)·L2(권장) 기준으로 채점.
- 점수는 “해당 챕터의 요구사항 중 충족 비율”에 대한 전문가 추정이며, 근거를 코드 위치로 명시.
- 보조 기준: **12-Factor App**(운영 성숙도), **SOC 2 Common Criteria**(조직 통제), **NIST SP 800-57**(키 관리).

## 3. 강점 (그대로 유지·계승할 것)

- **암호 프리미티브 선택이 견실** — AES-256-GCM(인증 암호), HMAC-SHA256, `timingSafeEqual`(타이밍 공격 방어), CSPRNG(`randomBytes`).
- **불변식이 명확하고 테스트됨** — 단일유효토큰을 원자적 CAS로 보장, 권한 이전/동시성/위조/교차카피까지 29개 테스트.
- **계약 중심 모듈 설계** — `Store` 인터페이스로 영속성 교체 지점이 이미 분리됨(Postgres/Redis 어댑터를 끼우면 됨).
- **CEK를 컨테이너에서 분리** — 서버 권한이 복호화를 통제하는 올바른 방향.
- **입력 검증 일관성** — 모든 경계에서 Zod 파싱.

이 토대는 버리지 않고 그 위에 프로덕션 계층을 쌓으면 됩니다.

## 5. P1 — 출시 전 필수

- **결제 실연동** — PG(스트라이프/아임포트) 연동, 멱등키, 웹훅 서명 검증, 환불/분쟁. 현재 `/purchase`는 스텁.
- **감사 로그 영속화 + 변조 방지** — append-only 저장, 해시 체인/서명, 보존 정책, 내보내기.
- **관측성** — 구조화 로깅(요청ID), 메트릭(Prometheus), 분산추적(OpenTelemetry), 에러추적(Sentry), 알림(SLO 기반).
- **입력·자원 강화** — 업로드 MIME/크기 제한·AV 스캔, 플랫폼 포워더 **SSRF 방어**(고정 allowlist), 출력 인코딩 점검.
- **배포 표준화** — Dockerfile, IaC(Terraform), **graceful shutdown**, readiness/liveness 분리, 12-Factor 설정.
- **세션·토큰 강화** — 서버측 `exp` 강제 옵션, 디바이스 바인딩, 토큰 폐기(revocation) 목록.
- **백업/복구** — DB 백업·PITR, 마이그레이션(분리된 스키마 버전), DR 절차.

## 7. 목표 아키텍처 (참고)

```text
            ┌── WAF / CDN ──┐
   client → │  TLS 종단(LB)  │ → API (stateless, N replicas, @crypton/server)
            └───────────────┘        │  인증(JWT/세션) · 레이트리밋 · helmet
                                      ├── Postgres   (titles/copies/entitlements/audit)
                                      ├── Redis      (토큰 CAS · 세션 · 알림 pub/sub)
                                      ├── KMS/HSM    (마스터키 · CEK 봉투암호화)
                                      └── Object Store(원본 암호문 페이로드)
```

현재 단일 프로세스·인메모리 구조에서 위 구조로 가는 핵심은 **`Store` 인터페이스 구현 교체 + 무상태 API + Redis 원자 회전**입니다(설계상 이미 분리되어 있어 침습이 적음).
