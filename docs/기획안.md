## 0. 개요
- **무엇:** 문서를 *깨질 수 있는 암호*가 아니라, **서버가 통제하는 회전 토큰**으로 보호한다. 열 때마다 토큰이 바뀌고, **유효 토큰은 한 시점에 1개**라 무단 재배포가 무력화된다.
- **핵심 가치:** ① 한 번 풀려도 영구 무력화되지 않음(서버가 권한의 단일 출처) ② 복잡한 DRM 불필요 ③ 오프라인 열람 가능 ④ 전자서점(결제·판매)·클라우드 모델 내장.
- **본질:** 이건 *암호 제품*이 아니라 **접근·배포 통제 제품**이다(§6 위협모델 참고).

## 1. 설계 원칙
1. **서버 = 권한의 단일 출처(Single source of truth for access).** 열람 가부는 항상 서버 토큰 인증으로 결정.
2. **단일 유효 토큰 불변식(single-live-token).** 문서당 유효 토큰은 정확히 1개. 회전은 **원자적(atomic CAS)**.
3. **무결성은 고유값(해시)로.** 토큰 서명·검증에 불변 고유값 사용.
4. **뷰어는 프로토콜을 강제한다.** 일반 PDF 뷰어가 아닌 **전용 뷰어**가 열람 시 토큰 교환을 수행, *열람 개시 명령*을 받아야만 렌더.
5. **정직한 위협모델.** 화면 캡처 등 아날로그 홀을 과장하지 않는다 → 포렌식 워터마크로 *억제*.

## 2. 시스템 구성
| 컴포넌트 | 역할 |
|----------|------|
| **server** (보안 서비스 제공 서버, 도2·청구항8~10) | 토큰 발급·인증·**회전**, 단일유효토큰 원자 보장, 결제 게이트, 알림, 감사 로그 |
| **viewer** (전용 뷰어) | 열람 시도 → 서버에 토큰 전송(API) → *열람 개시 명령 + 새 토큰* 수신 → 토큰 갱신 후 렌더. **오직 이 뷰어만** 문서를 연다 |
| **platform** (전자서점, PF) | 웹/앱 마켓플레이스: 문서 등록·구매·다운로드. 문서=도서, 가격 표시 |
| **core** (공유) | 토큰 모델·암호·문서 컨테이너 포맷·API 타입(서버/뷰어/플랫폼 공통 계약) |
| **document container** (`.crypton`) | AES-GCM 암호화된 페이로드 + 매니페스트(고유값·토큰 슬롯·서버 엔드포인트) |

## 3. 핵심 플로우
### 3.1 구매 → 다운로드 (청구항 1·3)
```
client → platform: 문서 구매(결제)            // 청구항3: 결제 완료가 전송 선행조건
server: 문서에 고유값(hash) + 제1토큰(T1) 할당·보관
server → client: 고유값+T1 포함 문서(.crypton) 전송 (AES-GCM)
```
### 3.2 열람 → 토큰 회전 (청구항 1·2·4) — 발명의 심장
```
viewer(open 시도) ──API(T1)──▶ server
server: 보유 토큰과 T1 비교 인증 (고유값으로 서명 검증)
  ├─ 성공 → 제2토큰(T2) 생성(T1≠T2), 보유토큰 T1→T2 원자 교체
  │        server ──(열람개시 명령 + T2)──▶ viewer → 문서 토큰 T1→T2 갱신 → 렌더
  └─ 실패 → 열람개시 명령 미전송(또는 불가) → 뷰어 열람 차단
```
### 3.3 공유 시 권한 이전 (도4·청구항 5~7) — 무단배포 무력화
```
A(T1) ──공유──▶ B(T1)
B open(T1) → server: T1 인증 OK → T2 발급(보유=T2) → B 렌더
A re-open(T1) → server: 보유 T2 ≠ T1 → 인증 실패 → A 열람 불가
(옵션) B 열람 시 A 온라인 → A에 알림 / A 열람 중지
```
### 3.4 오프라인 (【0078】)
- 열람 *시점*에만 온라인이면 토큰 갱신 → 이후 **오프라인 열람 가능**. 다음 open 시 재검증.
- 설계 결정: **오프라인 유예 창(grace window)** 정책 필요(§12).
### 3.5 클라우드 (도5·청구항 외)
- 공용 계정으로 문서 공유하되 **동시 1단말**만. 이용마다 토큰 회전. 클라우드 이용 = **프리미엄 과금**.

## 4. 데이터 모델 & 토큰
### 4.1 핵심 테이블
```
documents      { id(고유값=hash, PK), title, owner_id, status,
                 current_token_id, current_token_sig, updated_at }
token_log      { doc_id, token_id, issued_to(session/device), issued_at, replaced_at }  // 감사·포렌식
entitlements   { user_id, doc_id, source(purchase/cloud), granted_at }
sessions       { id, user_id, device, last_seen }
```
### 4.2 토큰 구조 (【0061】 기반)
```jsonc
{
  "doc": "<고유값 hash>",      // 불변
  "tid": "<nonce, uuidv7>",    // 회전마다 새로 (T1≠T2)
  "iat": 1719200000,
  "exp": 1719203600,           // 유효시간
  "perm": ["read"],            // 권한
  "sid": "<session/process>",  // 프로세스 식별
  "sig": "<HMAC/Ed25519(고유값 키)>"  // 무결성·출처
}
```
- **회전 = 새 `tid` 발급 + `documents.current_token_id` 원자 교체.** 동시성은 **Redis 분산락 / DB compare-and-swap** 으로 단일유효 보장.
- 재생공격 방지: `tid` 일회성 + `exp` + 사용 즉시 무효화.

## 5. 아키텍처
```
[ platform 웹/앱 ] ──구매/다운로드──▶ [ server: API ]
                                         │  토큰 권한(단일출처)
[ viewer (전용) ] ──open: API(token)──▶  ├─ Token Authority (발급/인증/회전, atomic)
       ▲  열람개시+새토큰                ├─ Payment / Entitlement
       └──────────────────────────────  ├─ Notify (websocket/push)
                                         └─ Store: Postgres + Redis(토큰상태/락) + Audit log
```
- 서버 중심(server-authoritative). 뷰어는 *얇은 신뢰*(렌더 + 프로토콜), 권한 판단은 전부 서버.
- 단일유효토큰의 **원자성**이 정확성의 핵심 → Redis Lua/`WATCH`-MULTI 또는 Postgres `UPDATE ... WHERE current_token_id = $expected`.

## 6. 보안 위협 모델 (정직하게)
**막는 것 ✅**
- 유효 토큰 없는 **무단 열람**, **무단 재배포**(단일유효토큰), 즉시 **권한 회수/회전**, 전체 **감사 추적**.
- PDF 암호처럼 *한 번 풀리면 끝*인 영구 무력화를 방지(서버가 매 열람을 통제).

**완전히 못 막는 것 ⚠️ (과장 금지)**
- 렌더된 화면의 **스크린샷/촬영**, 뷰어 역공학 후 **메모리 추출**(특히 웹 뷰어).
- **억제책:** 사용자별 **포렌식 워터마크**(유출 추적), **하드닝된 네이티브 뷰어**, OS 레벨 스크린샷 차단(부분적), 짧은 토큰 수명·세션 바인딩.
- 결론: crypton은 **"인가된 열람 + 배포 통제"** 를 제공한다. *절대적 복제 방지 DRM이 아니다* — 이 경계를 마케팅·계약에 명확히.

## 7. 기술 스택
| 레이어 | 선택 | 이유 / 대안 |
|--------|------|------------|
| **server** | **TypeScript (Bun/Node) + Fastify** · **Postgres** · **Redis** | 빠른 반복·생태계. 토큰 회전 원자성은 Redis. *(스케일 시 Token Authority만 **Go/Rust**로 분리 가능)* |
| **core** | **TypeScript** (Zod 스키마, 토큰/컨테이너/API 타입) | 서버·뷰어·플랫폼 공통 계약 |
| **viewer** | **데스크톱: Electron + PDF.js** · **모바일: React Native** · (웹: PDF.js, 보호 약함) | 본인 RN/Electron 보유. 전용 뷰어로 프로토콜 강제 *(강보안 필요 시 네이티브 Swift/Kotlin 뷰어)* |
| **platform** | **Next.js / React** (+ 뷰어 앱 재사용) | 전자서점 웹 + 앱 |
| **crypto** | **AES-256-GCM**(컨테이너/전송) · **Ed25519/HMAC-SHA256**(토큰 서명, 고유값 키) | 특허는 AES-128 예시(비한정) → 256-GCM 권장 |
| **암호 키 관리** | 서버 KMS / HSM(추후) | 서명키 보호 |

> 계약(contract)은 **core의 토큰·컨테이너·API 타입**. 뷰어/서버 언어가 달라도 이 계약으로 상호운용.

## 8. 결제 · 커머스 (전자서점)
- 문서=상품. **결제 완료가 다운로드 선행조건**(청구항3). 결제(PG)·구독·클라우드 프리미엄 과금.
- 출판사/저자 대시보드(P2): 업로드, 가격, 열람 통계(감사 로그 기반).

## 9. 확장성
- **문서 타입 확장:** PDF → epub, office, 이미지(컨테이너 포맷이 추상화).
- **권한 모델 확장:** read → print/annotate/expire, 기간제·횟수제 라이선스(토큰 `perm` 확장).
- **정당한 다중기기**(본인 소유 N대) vs 공유 구분: entitlement + 동시세션 정책.
- **클라우드/엔터프라이즈:** SSO, 조직 정책, 온프레미스 서버.

## 10. 로드맵
- **P0 (MVP, 발명 코어 증명):** server 토큰 권한(발급/인증/회전, **원자 단일유효**) + 데스크톱 viewer(Electron+PDF.js)로 open→토큰→열람개시→회전 + 최소 platform(업로드/목록/다운로드). → *공유 시 권한 이전*을 데모.
- **P1:** 결제·전자서점, 모바일 viewer(RN), `.crypton` 컨테이너, 변위 보유자 알림.
- **P2:** 클라우드 단일세션, 오프라인 유예, **사용자별 워터마크**, 감사·분석, 출판사 대시보드.
- **P3:** 스케일(Go/Rust 토큰 코어·HA), 엔터프라이즈, 문서타입 확장, 스크린샷 억제 강화.
